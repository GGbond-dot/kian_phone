# KHUP 行为线 MVP — 技术方案总览

## 0. 阅读说明

这是 KHUP（异常值认知教练 App）"行为线 MVP"的技术方案，分为 6 个文件。

**每个模块文件自包含**，可以独立交给不同 AI 实现。模块之间通过明确的接口契约通信，互不依赖内部实现。

```
00_总览.md            ← 你正在读
01_数据层.md           Module 1: DB schema, Migration, Entities, DAOs
02_AI生成层.md         Module 2: Prompt Policy, Generators
03_Repository层.md     Module 3: Repositories, 触发链路
04_UI层.md             Module 4: TodayScreen, HistoryScreen, 路由
05_验收清单.md         端到端验收（所有模块完成后跑一遍）
```

**实现顺序**：

```
Module 1 (数据层)              ← 必须先做
        ↓
   ┌────┴─────┐
   ↓          ↓
Module 2     Module 3        ← 可以并行
   ↓          ↓
   └────┬─────┘
        ↓
Module 4 (UI)                  ← 依赖 Module 3 的 Repository 接口
        ↓
05 端到端验收
```

## 1. 项目背景（让 AI 理解产品哲学，避免生成"普通防沉迷工具"）

KHUP 是一个 Android 个人应用，作者 Kian 自用（Xiaomi 14）。

**它不是**：防沉迷工具、屏幕时间提醒器、日程管家。

**它是**：基于塔勒布"反脆弱 / 黑天鹅 / 可选性"思想的"异常值认知教练"。它帮助用户识别自己被算法和习惯推回的"回归值生活"，并提出**低成本、正期望、可拒绝**的"异常值"行动。

### 1.1 核心三件套（每条建议都必须满足）

| 原则 | 含义 | 允许 / 禁止 |
|---|---|---|
| **低成本** | 用户失败也无损失 | ✅ 多走 10 分钟、写一句、看一篇短文 ❌ 跨城旅行、翘课、大额消费、危险行为、高压力社交 |
| **正期望** | 必须有可写出来的潜在收益 | ✅ 新信息 / 新感受 / 新视角 / 新行动可能 |
| **可拒绝** | 用户能"接受 / 换一个 / 不适合"，不命令 | UI 必须保留拒绝路径 |

### 1.2 不衡量执行原则（重要）

> **建议被接受即终态，不追问是否真的去做。**

系统**不**：
- 打分
- 计算"完成率"
- 提醒用户兑现
- 把用户分类成"自律 / 不自律"

这条原则是为了避免把"教练"变成"考核器"，违背可选性哲学。这也是为什么 UI 层只有 3 个反馈按钮（接受 / 换一个 / 不适合），**没有"完成"按钮**。

### 1.3 文案风格

锋利但不羞辱。

✅ "算法没有挑战你，它只是在喂养你。"
❌ "你太废了，又刷了一天垃圾内容。"

## 2. 行为线 MVP 范围

### 2.1 做什么

跑通这条最小闭环：

```
用户主动输入处境
  → 落 Event(USER_REPORT)
  → RegressionPatternGenerator 识别回归值（写 AttentionAnomaly）
  → AnomalySuggestionGenerator 生成异常值建议（写 AnomalySuggestion）
  → 显示在 TodayScreen 卡片
  → 用户点击 [接受][换一个][不适合]
  → 写 UserFeedback + 更新 AnomalySuggestion.status
  → 下次生成时把 7 天 feedback 作为 prompt 上下文
```

### 2.2 不做什么（明确排除）

- ❌ 信息线 / OCR / 屏幕采样 / 悬浮窗
- ❌ 自动定位 / 路线识别
- ❌ 通知 / 用机统计 UI 重做（保留旧 DashboardScreen 文件，但路由切到新 TodayScreen，下个迭代再删旧文件）
- ❌ 数据导出 / 清空的实际功能（设置页只占位）
- ❌ 模型微调 / 反馈权重学习
- ❌ Phase 0 安全底盘（destructive migration 修复、log 脱敏，仅在本方案明确指定的几处做，全面修复留下迭代）

## 3. 模块共享契约（每个模块都会引用）

### 3.1 数据库版本

`v10 → v11`，**禁止 destructive migration**。

### 3.2 EventType 枚举（Module 1 维护）

```kotlin
enum class EventType {
  // 现有项保留 ...
  USER_REPORT,    // 新增：用户主动输入的行为处境
}
```

### 3.3 关键实体引用（详见 Module 1）

```
Event              — 不可变事实（含 USER_REPORT）
AttentionAnomaly   — 回归值模式（v11 扩展字段）
AnomalySuggestion  — 异常值建议（由 daily_tasks 迁移而来）
UserFeedback       — 通用反馈（新表）
```

### 3.4 Repository 接口（Module 3 提供，Module 4 调用）

完整接口见 `03_Repository层.md`，关键方法：

```kotlin
interface BehaviorReportRepository {
    suspend fun submit(text: String): String  // 返回 eventId
}

interface AnomalySuggestionRepository {
    fun observeTodayPending(): Flow<AnomalySuggestion?>  // TodayScreen 卡片观察
    suspend fun accept(id: Long)
    suspend fun postpone(id: Long)                        // 触发立即重新生成
    suspend fun reject(id: Long, reason: String?)
    fun observeRecentByStatus(status: String): Flow<List<AnomalySuggestion>>  // HistoryScreen
}

interface UserFeedbackRepository {
    suspend fun record(targetType: String, targetId: Long, feedbackType: String, reason: String? = null)
}
```

### 3.5 Generator 接口（Module 2 提供，Module 3 调用）

```kotlin
interface RegressionPatternGenerator {
    suspend fun analyzeUserReport(eventId: String): Long?  // 返回 patternId，未识别返回 null
}

interface AnomalySuggestionGenerator {
    suspend fun generateForPattern(patternId: Long, regenerationCount: Int = 0): Long?  // 返回 suggestionId
}
```

## 4. 全局约束（所有模块都必须遵守）

### 4.1 隐私 / 日志

- LLM 输出**不写全文 logcat**。即使 DEBUG build 也只能写 redact 后摘要（< 100 字）。
- 用户输入的 `USER_REPORT.text` 在送 prompt 前必须经 `PromptRedactor` 脱敏（人名、地名等）。

### 4.2 Migration

- 每个新增列必须有 `DEFAULT` 值（避免 NOT NULL 写入失败）。
- 必须提供 migration 测试，验证 v10 → v11 数据不丢失。
- 所有 `ALTER TABLE ... ADD COLUMN` 必须写在 `Migration_10_11.kt` 里，不能依赖 Room 的 `fallbackToDestructiveMigration()`。

### 4.3 LLM 输出验证

所有 LLM 生成的 suggestion 必须：

```
costLevel == "LOW"           # 强制；其他值丢弃重试一次
expectedUpside.isNotBlank()  # 必填
actionText.length <= 200     # 防止生成长篇
whyText.length <= 200
```

不满足时 fallback 到静态温和提醒（见 Module 2）。

### 4.4 文件长度

新增 Compose UI 文件**单文件目标 < 200 行**。Service 文件**单文件目标 < 300 行**。超出时拆分。

## 5. 模块间的失败/降级策略

| 场景 | 策略 |
|---|---|
| LLM 失败 / JSON 解析失败 | Fallback 静态文案（见 Module 2 fallback 常量），不暴露给用户 |
| RegressionPatternGenerator 未识别出 pattern | UserReport 仍落库；不生成 suggestion；TodayScreen 显示空态引导文案 |
| 24h 内同 patternKey 已有 PENDING/ACCEPTED/REJECTED | 不再生成新 suggestion；TodayScreen 显示空态引导文案 |
| 用户连续点"换一个"超过 5 次 | 不限制，但 prompt 会带 regenerationCount，期望 LLM 生成不同角度 |

**空态统一引导文案**：

> 今天还没有可说的回归值信号。要不要写一段你现在的处境？
