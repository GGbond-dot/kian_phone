# Module 9 — 补丁：建议多样性 + AI 对话深度

## 背景

当前的两个体验问题：
1. 异常值建议"感觉很单一"——总是类似"去走 10 分钟"，没有切换领域
2. AI 聊天"设计得不够好"——像在跟陌生人对话，不像认识你的教练

根本原因：LLM 每次生成时上下文太稀疏，不知道：
- 这周已经给了哪些领域的建议（Domain 历史）
- 用户对哪类建议接受/拒绝过（Feedback 历史）
- 用户最近的整体行为模式是什么（Pattern 摘要）

本补丁分三部分修复：
- A. Domain 轮转：避免重复给同一领域的建议
- B. Prompt 推理深度：把"三件套"从标签升级为推理框架
- C. AI 聊天注入 KHUP 用户上下文

---

## Part A：Domain 轮转（避免建议总是同一类型）

### A.1 什么是 suggestionDomain

`AnomalySuggestion.suggestionDomain` 有 6 个值：

```
BEHAVIOR   — 行为路径（走路、换空间、停下来）
INFORMATION — 信息偏离（看一篇反向文章、关掉某类内容）
SOCIAL     — 社交维度（给某人发一条消息、不同的对话方式）
SPACE      — 空间变化（去一个平时不去的地方）
BODY       — 身体感知（拉伸、冷水、呼吸）
CREATION   — 创作/输出（写一句、画一笔、录音）
```

MVP 大量输出 BEHAVIOR 是因为 prompt 里没有引导 LLM 考虑其他领域。

### A.2 新增：获取 7 天 Domain 分布

在 `AnomalySuggestionDao` 中加：

```kotlin
@Query("""
    SELECT suggestionDomain, COUNT(*) as count
    FROM anomaly_suggestion
    WHERE createdAt >= :sinceMs
    GROUP BY suggestionDomain
""")
suspend fun getDomainCountsSince(sinceMs: Long): List<DomainCount>

data class DomainCount(val suggestionDomain: String, val count: Int)
```

### A.3 修改 AnomalySuggestionGeneratorImpl

在组装 prompt 之前，加 domain 统计逻辑：

```kotlin
// 取 7 天 domain 分布
val sevenDaysAgo = now - 7 * 24 * 3600 * 1000L
val domainCounts = suggestionDao.getDomainCountsSince(sevenDaysAgo)
    .associate { it.suggestionDomain to it.count }

// 找出本周已用 >= 3 次的领域（过度使用）
val overusedDomains = domainCounts.filter { it.value >= 3 }.keys

// 推荐领域 = 全部 - 过度使用的
val allDomains = listOf("BEHAVIOR", "INFORMATION", "SOCIAL", "SPACE", "BODY", "CREATION")
val suggestedDomains = (allDomains - overusedDomains).takeIf { it.isNotEmpty() } ?: allDomains

val domainHint = buildString {
    if (overusedDomains.isNotEmpty()) {
        append("本周已频繁出现的建议领域（请尽量避免）：${overusedDomains.joinToString()}。\n")
    }
    append("优先考虑的建议领域：${suggestedDomains.joinToString()}。\n")
    append("各领域含义：BEHAVIOR=行为路径, INFORMATION=信息偏离, SOCIAL=社交维度, SPACE=空间变化, BODY=身体感知, CREATION=创作输出。")
}
```

把 `domainHint` 加入 prompt 输入 JSON：

```json
{
  "domain_guidance": "本周已频繁出现 BEHAVIOR(5次)，优先考虑 INFORMATION / SPACE / CREATION...",
  "pattern": { ... },
  "recent_feedbacks": [ ... ]
}
```

### A.4 修改 Prompt：明确要求多样化

在 `BEHAVIOR_LINE_SYSTEM_PROMPT` 里加一段：

```
【建议多样性】
- 六个建议领域（BEHAVIOR / INFORMATION / SOCIAL / SPACE / BODY / CREATION）是平等的
- 不要总是推荐走路、散步等行为路径
- 输入中会告诉你本周哪些领域已过多，请优先选择其他领域
- "走 10 分钟" 是 BEHAVIOR；"看一篇你不同意的文章" 是 INFORMATION；"给一个久没联系的人发一句话" 是 SOCIAL
- 异常值的价值在于偏离惯性，不同领域的偏离是不同维度的选择权恢复
```

## Part B：Prompt 推理深度（从标签到框架）

### B.1 当前问题

当前 prompt 的三件套是"输出校验标签"：

```
✓ costLevel = "LOW"
✓ expectedUpside = 非空
✓ 可拒绝
```

LLM 先想一个建议，再贴上这三个标签。结果是建议本身不一定真的是塔勒布式思考。

### B.2 改造：把三件套变成推理步骤

修改 `BEHAVIOR_LINE_SYSTEM_PROMPT`，把三件套改成**思考流程**：

```
【生成异常值建议的推理顺序】

不要先想"我给什么建议"，而是按以下顺序推理：

第一步：识别"回归值"
  - 用户描述的情境里，哪个行为/选择是"算法或惯性的最低阻力选项"？
  - 这个默认选项让用户损失了什么选择权？
  - 举例：会议结束→疲劳→刷手机，损失的是"对疲劳状态如何处理"的选择权

第二步：寻找"最小偏离点"
  - 不是让用户改变人生，而是在哪个小节点上插入一个不同的选择？
  - 偏离成本要极低（失败了也无损失）
  - 偏离方向要有潜在收益（不是随机作死，是有正期望）

第三步：确认可拒绝性
  - 这个建议是否容易拒绝、推迟或替换？
  - 用户拒绝了，是否仍能感受到一点点认知摩擦？（好的建议即使被拒，也让用户停顿了一秒）

第四步：选择领域（重要）
  - 基于 domain_guidance 和用户情境，这个偏离应发生在哪个维度？
  - 不要默认选 BEHAVIOR，先问：这个情境里 INFORMATION / SOCIAL / SPACE / BODY / CREATION 有没有更自然的偏离点？

第五步：锻造文案
  - actionText：具体、可执行，像一个真实的指令，不是原则陈述
  - whyText：解释"为什么这是回归值"，不是解释建议本身
  - 锋利但不羞辱
```

### B.3 AI 聊天 Prompt 深度（针对自由对话）

AI chat 的系统提示词（当前可能是通用 KhupPromptPolicy）里加：

```
【作为认知教练的对话风格】

你不是问答机器，而是认知教练。对话时：

1. 先倾听，不要急于给建议
   - 用户说一件事，先问一个问题帮他说清楚："你说太累了——是身体累还是心里累？"
   - 不要在用户说完第一句话就列举三条建议

2. 识别回归值逻辑，不是识别"问题"
   - 不要说"你遇到了X问题"
   - 要说"我注意到一个模式：每次Y发生，你似乎就会Z"

3. 塔勒布式思考，不是正面激励
   - 不要说"你可以做到的！"
   - 要说"如果这条路成本极低且你只是没想到——试试看有什么损失？"

4. 对拒绝保持好奇，不是说服
   - 用户说"不想做"：先接受，再问"是什么让这个成本变高了？"
   - 不要说"但是你应该试试"

5. 保持简短
   - 每次回复不超过 3 段
   - 不列编号清单（编号清单是培训材料的语气，不是教练的语气）
```

## Part C：AI 聊天注入 KHUP 用户上下文

### C.1 问题

当前 AI tab 每次是空白开始，AI 不知道：
- 用户最近在关注什么回归值模式
- 哪类建议接受过，哪类拒绝过
- 最近的情绪/处境倾向

### C.2 方案：KhupContextSummarizer

新建 `core/ai/KhupContextSummarizer.kt`：

```kotlin
/**
 * 从 DB 生成一段 150 字以内的用户上下文摘要，注入 AI 对话。
 * 不发送原始数据，只发送聚合后的模式描述。
 */
@Singleton
class KhupContextSummarizer @Inject constructor(
    private val anomalyDao: AttentionAnomalyDao,
    private val suggestionDao: AnomalySuggestionDao,
    private val feedbackDao: UserFeedbackDao,
    private val eventDao: EventDao,
) {
    suspend fun buildUserContext(): String {
        val now = System.currentTimeMillis()
        val sevenDaysAgo = now - 7 * 24 * 3600 * 1000L

        // 1. 最活跃的回归值模式（最多 3 条）
        val activePatterns = anomalyDao.getActivePatternsSince(sevenDaysAgo, limit = 3)

        // 2. 建议接受/拒绝分布
        val recentSuggestions = suggestionDao.getRecentWithStatus(sevenDaysAgo)
        val acceptedCount = recentSuggestions.count { it.status == "ACCEPTED" }
        val rejectedCount = recentSuggestions.count { it.status == "REJECTED" }
        val domainCounts = recentSuggestions.groupBy { it.suggestionDomain }
            .mapValues { it.value.size }

        // 3. 最常拒绝的原因（取 user_feedback.reason 非空的前 3 条）
        val rejectedReasons = feedbackDao.recentByTargetType("SUGGESTION", sevenDaysAgo, 20)
            .filter { it.feedbackType == "REJECT" && !it.reason.isNullOrBlank() }
            .take(3)
            .mapNotNull { it.reason }

        // 4. 组装摘要
        return buildString {
            appendLine("【用户近期状态（7天内，仅供参考，勿直接引用）】")
            if (activePatterns.isNotEmpty()) {
                appendLine("主要回归值模式：${activePatterns.joinToString("；") { it.title }}")
            }
            if (recentSuggestions.isNotEmpty()) {
                appendLine("建议：共${recentSuggestions.size}条，接受${acceptedCount}条，拒绝${rejectedCount}条")
            }
            if (domainCounts.isNotEmpty()) {
                val domainSummary = domainCounts.entries.sortedByDescending { it.value }
                    .joinToString("、") { "${it.key}(${it.value}次)" }
                appendLine("建议领域分布：$domainSummary")
            }
            if (rejectedReasons.isNotEmpty()) {
                appendLine("常见拒绝原因：${rejectedReasons.joinToString("；")}")
            }
        }.trim().take(400)   // 硬截断，不要让 context 太长
    }
}
```

**注意**：这个摘要进入 prompt 的**系统消息**，不是用户消息。用户不会看到这段话。

### C.3 AiViewModel 注入上下文

```kotlin
@HiltViewModel
class AiViewModel @Inject constructor(
    private val aiContextBridge: AiContextBridge,
    private val contextSummarizer: KhupContextSummarizer,
    // ... 其他依赖
) : ViewModel() {

    init {
        viewModelScope.launch {
            // 1. 生成用户上下文摘要（每次开新会话时）
            val userContext = try {
                contextSummarizer.buildUserContext()
            } catch (t: Throwable) {
                ""  // 失败了也没关系，降级为无上下文
            }

            // 2. 构建注入了用户上下文的系统提示词
            val systemPrompt = buildSystemPrompt(userContext)

            // 3. 如果有来自"不适合"的预填消息，发出去
            val pending = aiContextBridge.consumePending()
            if (pending != null) {
                startNewSessionWithContext(systemPrompt, pending.message, pending.suggestionId)
            } else {
                startNewSessionWithContext(systemPrompt, null, null)
            }
        }
    }

    private fun buildSystemPrompt(userContext: String): String {
        return buildString {
            append(KhupPromptPolicy.AI_CHAT_SYSTEM_PROMPT)  // 基础 prompt（见 B.3）
            if (userContext.isNotBlank()) {
                append("\n\n")
                append(userContext)
                append("\n\n以上是系统从用户数据中提取的模式摘要，仅供你理解用户背景。对话中不要直接朗读这些数据，而是自然地把它们融入你对用户的理解中。")
            }
        }
    }
}
```

## 补丁验收

### 建议多样性

- [ ] 连续提交 6 次不同处境，观察 `suggestionDomain` 字段分布：不应全部为 BEHAVIOR
- [ ] 在 `anomaly_suggestion` 表手动把同一 domain 插入 3 条（测试数据），再提交一次 → 新建议 domain 应切换
- [ ] 建议文案不再是"走 10 分钟"的固定模板；actionText 在不同场景有明显差异

### Prompt 推理深度

- [ ] 提交"我刚刷了 2 小时微博"的处境，建议 domain 应优先 INFORMATION，而不是 BEHAVIOR
- [ ] 提交"我睡前又打开了抖音"，建议应识别到"睡前算法使用"回归值，而不是泛化的"去走走"
- [ ] 建议的 whyText 应描述"为什么这是回归值"，不是"为什么这条建议好"

### AI 对话质量

- [ ] 打开 AI tab，发"我最近总是开完会就想刷手机"
  - AI 不应立刻列出三条建议
  - AI 应先问一个问题（澄清情境）
- [ ] AI 的回复不使用编号列表（1. 2. 3. 格式）
- [ ] AI 能说出类似"我注意到你最近有这个模式..."（如果 DB 里有数据）
- [ ] AI 聊天后切换到 Today tab，MiniObservationCard 的数据不变（AI 上下文注入不影响其他功能）

### 隐私

- [ ] `KhupContextSummarizer.buildUserContext()` 的输出最多 400 字符（grep 验证）
- [ ] 用户上下文摘要不包含原始 `text` 内容（只有模式名称、数量，不转发原文）
