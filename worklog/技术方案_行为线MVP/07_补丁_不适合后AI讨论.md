# Module 7 — 补丁：[不适合] 后进入 AI 讨论

## 背景

当前 [不适合] 按钮的流程：建议标记 REJECTED → 卡片消失 → 结束。

这是死路。用户想要的是：拒绝之后能和 AI 聊"为什么不合适"，继续分析或换一个角度。

本补丁在不改动 TodayScreen 主布局的前提下，给 [不适合] 增加"和 AI 聊聊"的出口。

---

## 1. 流程变化

### 现状
```
点 [不适合] → dialog（可选填 reason）→ [确定] → REJECTED → 卡片消失
```

### 目标
```
点 [不适合] → dialog（可选填 reason）→ [算了跳过] 或 [和 AI 聊聊 →]
                                            ↓                    ↓
                                        REJECTED             REJECTED
                                        卡片消失       +     跳到 AI tab
                                                       +     自动发送预填上下文
```

两条路都会写 REJECTED + UserFeedback，区别只是后者多一个"跳转到 AI 聊天"的动作。

## 2. 新增：AiContextBridge

文件：`core/ai/AiContextBridge.kt`

```kotlin
/**
 * 内存中的单向桥：Today → AI tab 的上下文传递。
 * 使用单例内存状态，避免导航参数的 URL 编码问题。
 */
@Singleton
class AiContextBridge @Inject constructor() {

    private val _pending = MutableStateFlow<String?>(null)

    /**
     * TodayScreen 写入：用户点"和 AI 聊聊"时调用。
     * 下次 AiViewModel 初始化时会消费这条消息。
     */
    fun setPending(context: String) {
        _pending.value = context.trim().take(1000)
    }

    /**
     * AiViewModel 消费：进入 AI tab 时调用一次，随后清空。
     * 返回 null 表示没有待处理的上下文（正常打开 AI tab 时的情形）。
     */
    fun consumePending(): String? = _pending.getAndUpdate { null }
}
```

DI：在 `common/di/AiModule.kt`（或现有 Hilt 模块）中注册为 `@Singleton`（Hilt 自动处理，只需确保该类有 `@Singleton @Inject constructor`）。

### 3.2 新增方法：confirmRejectAndChat

```kotlin
/**
 * 用户在 reject dialog 选择"和 AI 聊聊"。
 * 写 REJECTED，构建上下文并推入 AiContextBridge，然后触发导航。
 */
fun confirmRejectAndChat(suggestionId: Long, reason: String?) {
    viewModelScope.launch {
        // 1. 写 REJECTED（和 confirmReject 一样）
        suggestionRepository.reject(suggestionId, reason)

        // 2. 构建预填上下文
        val suggestion = // 取 suggestion 的 title / actionText / whyText
        val context = buildAiContext(suggestion, reason)

        // 3. 推入桥
        aiContextBridge.setPending(context)

        // 4. 触发导航（通过 UiState 中的 navigation event）
        _uiState.update { it.copy(navigationEvent = NavigationEvent.GoToAi) }
    }
}

private fun buildAiContext(suggestion: AnomalySuggestion, reason: String?): String {
    val reasonText = if (reason.isNullOrBlank()) "未说明" else reason.trim()
    return """
        我刚拒绝了一条异常值建议，想和你聊聊。
        
        建议标题：「${suggestion.title}」
        建议内容：「${suggestion.actionText}」
        为什么给这条建议：「${suggestion.whyText}」
        
        我觉得不适合，原因：$reasonText
        
        帮我分析一下为什么不适合，或者换一个角度给我建议。
    """.trimIndent()
}
```

data class UiState(

val uiState by viewModel.uiState.collectAsState()
LaunchedEffect(uiState.navigationEvent) {

```kotlin
AlertDialog(
    title = { Text("这条不太合适？") },
    text = {
        Column {
            Text("说说为什么（选填）", style = MaterialTheme.typography.labelSmall)
            Spacer(Modifier.height(8.dp))
            OutlinedTextField(
                value = reason,
                onValueChange = { if (it.length <= 100) reason = it },
                placeholder = { Text("比如：太累了不想出门") },
                maxLines = 3,
                modifier = Modifier.fillMaxWidth(),
            )
        }
    },
    confirmButton = {
        Row(horizontalArrangement = Arrangement.spacedBy(8.dp)) {
            TextButton(onClick = { onDismiss() }) {
                Text("算了，跳过")
            }
            TextButton(onClick = { onRejectAndChat(reason) }) {
                Text("和 AI 聊聊 →")                    // ← 新增
            }
            // 保留原来的"确定（只拒绝，不聊）"
            Button(onClick = { onConfirmReject(reason) }) {
                Text("确定")
            }
        }
    },
    onDismissRequest = { onDismiss() },
)
```

**三个出口**：

| 按钮 | 行为 |
|---|---|
| 算了，跳过 | 关闭 dialog，不写 REJECTED（维持 PENDING） |
| 确定 | 写 REJECTED，卡片消失（原有行为不变） |
| 和 AI 聊聊 → | 写 REJECTED + 跳转 AI tab（新增） |

**"算了，跳过" 的行为**：关闭 dialog 后建议继续显示为 PENDING。这让用户"不小心点到"时有后悔机会，不会强制拒绝。

### 5.2 init 块消费预填上下文

```kotlin
init {
    val pending = aiContextBridge.consumePending()
    if (pending != null) {
        // 用户从 [不适合] 跳来，自动发送预填消息
        // 就像用户自己输入并点发送，完整走一遍 sendMessage 逻辑
        viewModelScope.launch {
            sendMessage(pending)
        }
    }
}
```

**关键**：`consumePending()` 是"读取并清空"，所以：
- 用户从 [不适合] 跳来：init 会消费这条消息，正常发送
- 用户平时打开 AI tab：`consumePending()` 返回 null，init 什么都不做
- 用户在 AI tab 内部重新进入（back stack 恢复）：已经清空，不会重复发送

## 6. 主导航：onNavigateToAi 回调

`TodayScreen` 需要一个 `onNavigateToAi: () -> Unit` 回调参数（如果已有类似参数直接复用）。

`MainScreen`（NavHost 入口）传入：

```kotlin
composable("today") {
    TodayScreen(
        onNavigateToAi = {
            // 切换底部导航到 AI tab
            selectedTab = Tab.Ai
            // 如果 AI 不在 back stack 里，navigate 到 ai 路由
            navController.navigate("ai") {
                popUpTo("today") { saveState = true }
                launchSingleTop = true
                restoreState = false    // 不恢复旧状态，让 AiViewModel 重新 init
            }
        },
        // ... 其他回调
    )
}
```

**注意 `restoreState = false`**：这确保每次从 [不适合] 跳转到 AI tab 时，`AiViewModel` 重新执行 `init`，从而消费 `AiContextBridge` 里的上下文。如果 restoreState=true，ViewModel 不会重新 init，预填消息就不会发送。

## 7. AI 的系统提示词补充

在 `KhupPromptPolicy.BEHAVIOR_LINE_SYSTEM_PROMPT` 或 AI chat 的系统提示词里补充一段（如果 AI chat 用不同 system prompt，改那个）：

```
【来自建议拒绝的对话上下文】
用户有时会从拒绝一条异常值建议后进入对话，消息会包含被拒绝建议的内容和原因。
这种情况下，你应当：
1. 先理解用户为什么拒绝（不要假设对方懒惰或抵触，可能有合理原因）
2. 分析这条建议是否真的不适合，还是只是时机/形式问题
3. 如果合理，提出一个不同角度的低成本、正期望、可拒绝的替代建议
4. 替代建议同样必须满足三件套（低成本 / 正期望 / 可拒绝）
5. 不要说"你应该接受建议"或"不接受是不好的"

这是一个讨论，不是强制执行。
```

## 8. 模块验收

- [ ] [不适合] 弹出的 dialog 有三个出口：算了跳过 / 确定（只拒绝）/ 和 AI 聊聊
- [ ] 点"和 AI 聊聊 →"：suggestion 变 REJECTED，自动跳到 AI tab
- [ ] AI tab 自动发出一条消息，包含被拒建议内容 + 用户填写的原因
- [ ] AI 的第一条回复针对被拒建议进行分析，而不是泛化问候
- [ ] 平时打开 AI tab（不从 [不适合] 跳）：不触发预填消息，正常空白对话框
- [ ] 用户在 AI 聊完后切回 Today tab，再次打开 AI tab：不重复发送预填消息（consumePending 已清空）
- [ ] 点"算了，跳过"：建议维持 PENDING，dialog 关闭，卡片仍显示
- [ ] `AiContextBridge` 在两次相邻拒绝中不串台（第一次消费后第二次才写入）
- [ ] 代码中不出现 `setPending` 和任何其他地方的 `consumePending` 调用，只有上面指定的调用点

## 9. 关联：讨论记录进历史建议

用户需要在"历史 → 建议"列表里能看到：这条被拒的建议，后来和 AI 聊了什么。

### 9.1 ChatSession 加 linkedSuggestionId

**检查 `ChatSession` entity 是否已有 `extraJson` 或 metadata 字段**：
- 有 → 把 `linkedSuggestionId` 序列化进去，**不需要新增列**
- 没有 → 在 migration 里加列：

```sql
-- 写入 Migration_11_12 或 Migration_12_13（取决于当前 DB 版本）
ALTER TABLE chat_sessions ADD COLUMN linkedSuggestionId INTEGER;
CREATE INDEX IF NOT EXISTS index_chat_sessions_linkedSuggestionId
  ON chat_sessions(linkedSuggestionId);
```

对应 entity 加字段：
```kotlin
val linkedSuggestionId: Long? = null,
```

### 9.2 AiContextBridge 扩展

```kotlin
@Singleton
class AiContextBridge @Inject constructor() {

    data class PendingContext(val message: String, val suggestionId: Long)

    private val _pending = MutableStateFlow<PendingContext?>(null)

    /** TodayViewModel 调用：同时传 context 和 suggestionId */
    fun setPending(message: String, suggestionId: Long) {
        _pending.value = PendingContext(message.trim().take(1000), suggestionId)
    }

    /** AiViewModel 消费，读取并清空 */
    fun consumePending(): PendingContext? = _pending.getAndUpdate { null }
}
```

`TodayViewModel.confirmRejectAndChat` 同步更新：

```kotlin
aiContextBridge.setPending(context, suggestionId)   // ← 原来只传 context，现在多传 id
```

### 9.3 AiViewModel 打标 ChatSession

```kotlin
init {
    val pending = aiContextBridge.consumePending()
    if (pending != null) {
        viewModelScope.launch {
            // 先创建/开启新 ChatSession，打上 linkedSuggestionId
            val sessionId = chatRepository.startNewSession(
                linkedSuggestionId = pending.suggestionId
            )
            // 再发预填消息
            sendMessage(pending.message, sessionId)
        }
    }
}
```

`ChatRepository.startNewSession` 需要支持可选的 `linkedSuggestionId` 参数（如果已有 `startNewSession()`，加一个带默认值的重载即可）。
