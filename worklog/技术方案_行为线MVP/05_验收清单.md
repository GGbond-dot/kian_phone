# 端到端验收清单

四个模块都完成后，跑这份清单做最终验证。每条都需在真机或模拟器上实际操作。

读本文件前需要先看 `00_总览.md`，了解产品哲学（特别是不衡量执行原则）才能正确解读每条验收。

---

## 1. 数据层验收（Module 1）

- [ ] 装一台已有 v10 数据的设备（或 backup-restore），升级到新版本
- [ ] 启动后 logcat 不出现 Room schema mismatch 错误
- [ ] 旧 daily_tasks 数据出现在新 `anomaly_suggestion` 表中，原 isDone=true 的现在 status='ACCEPTED'
- [ ] `attention_anomaly` 旧数据 firstSeenAt = lastSeenAt = createdAt
- [ ] 安装后能正常使用（不闪退，DB 可读写）

---

## 2. AI 生成层验收（Module 2）

- [ ] 提交一段处境（"我刚开完会，平时会直接回宿舍刷手机"）
- [ ] 几秒内生成一条建议，落库 status=PENDING
- [ ] 生成的建议 costLevel="LOW"，expectedUpside 非空，actionText/whyText 都 ≤ 200 字符
- [ ] logcat 中**完全找不到**完整的 LLM 输出（grep 一段 actionText 内容应只有 redact 摘要）
- [ ] 拔网情况下（API 通道失败）也能生成 fallback 建议（actionText/whyText 包含 fallback 常量内容）
- [ ] 提交相同处境第二次（24h 内），不生成新建议（数据库 anomaly_suggestion 计数不变）

---

## 3. Repository 层验收（Module 3）

- [ ] `submit` 调用立即返回，不卡 UI（用 `measureTimeMillis` 验证 < 100ms）
- [ ] 点击"接受"后，`anomaly_suggestion.status` = ACCEPTED，`user_feedback` 多一条 ACCEPT 记录
- [ ] 点击"换一个"后：
  - 旧 suggestion.status = POSTPONED
  - 新 suggestion 在几秒内出现，regenerationCount = 旧+1
  - 新 suggestion 的 actionText 与旧不同（LLM 应避免重复，靠 prompt 上下文）
- [ ] 点击"不适合"后，suggestion.status = REJECTED，`user_feedback` 多一条 REJECT 记录
- [ ] REJECTED 之后 24h 内**不再**自动生成同 patternKey 的新建议

## 5. 哲学一致性验收（最重要）

**重读 `00_总览.md` 第 1 节**，然后验证以下事实在产品中**不存在**：

- [ ] 任何位置都不出现"完成"按钮 / "已完成" 选项
- [ ] 任何位置都不出现"评分 / 打分 / 完成率 / 自律 / 不自律"等词
- [ ] 任何位置都不出现"streak / 连续 N 天 / 累计完成 X 次"等计数器
- [ ] 任何建议的 costLevel 都是 "LOW"，无 MEDIUM/HIGH
- [ ] 任何建议都包含 expectedUpside，且不是 placeholder 文字
- [ ] 任何建议都同时给出 actionText 和 whyText（不只有 actionText）
- [ ] 接受建议后不要求用户在某个时间反馈结果
- [ ] 拒绝建议（"不适合"）不需要必填理由

如有任何一条违反，**回头检查实现**。这些是产品差异化的红线，不是细节。
