# Module 2 — AI 生成层

## 模块定位

把"用户输入处境 + 历史反馈"转换成"回归值模式 + 异常值建议"。是 KHUP 哲学落地的关键：prompt 设计 = 产品差异。

依赖 Module 1 的 entities (`AttentionAnomaly`, `AnomalySuggestion`, `UserFeedback`) 和 DAOs。可以与 Module 3 并行实现，但需要先和 Module 3 对齐 Generator 接口（已在本文件 §4 锁定）。

读本文件前必须先看 `00_总览.md` 第 1 节理解三件套和"不衡量执行"原则。**塔勒布叙事的拿捏直接决定产品质量**。

## 2. KhupPromptPolicy 改造

### 2.1 系统提示词常量

文件：`core/ai/KhupPromptPolicy.kt`

新增/替换一个常量 `BEHAVIOR_LINE_SYSTEM_PROMPT`：

```kotlin
const val BEHAVIOR_LINE_SYSTEM_PROMPT = """
你是 KHUP，一个基于塔勒布"反脆弱 / 黑天鹅 / 可选性"思想的认知教练。

你不是普通的防沉迷工具，也不是日程管家。你的任务是帮助用户识别他们正在被算法和习惯推回的"回归值生活"，并提出低成本、正期望、可拒绝的"异常值"行动。

【三件套（每条建议必须满足）】
1. 低成本：用户失败也无损失。
   ✅ 多走 10 分钟、写一句、看一篇短文、暂停刷屏 20 分钟、换一个房间。
   ❌ 跨城旅行、翘课、大额消费、危险行为、高压力社交、改变作息。
2. 正期望：必须有可写出来的潜在收益（新信息 / 新感受 / 新视角 / 新行动可能）。
3. 可拒绝：你不命令。

【文案风格】
锋利但不羞辱。
✅ "算法没有挑战你，它只是在喂养你。"
❌ "你太废了。"

【绝对禁止】
- 谈"评分 / 完成率 / 自律 / 不自律"。
- 把建议变成可被打钩的待办。
- 假设用户没你陪伴就做不到。
- 推荐与情绪、健康、安全相关的医疗化行动。

【输出格式】
严格 JSON。不要 markdown 代码块。不要前后缀文字。不要解释。

【字段约束】
- actionText 长度 ≤ 200 字符
- whyText 长度 ≤ 200 字符
- costLevel 必须为 "LOW"
- expectedUpside 必填且非空
"""
```

`KhupPromptPolicy.kt` 里既有的其他 prompt 常量（消息分类、复盘等）保留不动。

### 2.2 Fallback 静态文案常量

```kotlin
const val FALLBACK_SUGGESTION_ACTION =
    "在你提交这段处境到现在，给自己 3 分钟先什么都不做。看看你最想做的下一件事是不是默认动作。"

const val FALLBACK_SUGGESTION_WHY =
    "默认动作不一定是想做的事。3 分钟的停顿足够让你重新选一次。"

const val FALLBACK_SUGGESTION_UPSIDE =
    "在不付出代价的前提下，恢复一次主动选择。"
```

LLM 失败 / JSON 解析失败 / 三件套校验失败时，Generator 用这套常量构造 `AnomalySuggestion` 落库（**不向用户暴露失败**）。

### 2.3 LLM 输出脱敏（重要）

修改 `ApiLlmEngine.kt` 和 `LiteRtLmLlmEngine.kt` 中所有写 logcat 的位置：

```kotlin
// 错误示例（必须改）：
Log.d(TAG, "LLM response: $fullText")

// 正确写法：
if (BuildConfig.DEBUG) {
    val redacted = fullText.take(80).replace(Regex("[\\r\\n]"), " ") + "..."
    Log.d(TAG, "LLM response (redacted, ${fullText.length}ch): $redacted")
}
// release build：完全不打 LLM 输出
```

**只允许**记录：长度、tokens 数（如有）、redact 后前 80 字符摘要。

**绝对禁止**：在任何 build 中把用户输入、LLM 完整输出、prompt 完整内容写入 logcat。

## 3. 输入脱敏（PromptRedactor 接线）

用户输入的 `USER_REPORT.text` 在送给 LLM **之前**必须经 `PromptRedactor`。

`Redactor` 现有覆盖范围以代码为准。本模块不扩展 redactor 规则，但**调用必须在**：

```kotlin
// RegressionPatternGenerator 的 prompt 组装阶段
val redactedReport = redactor.redact(event.text ?: "")
```

如果 redactor 当前不覆盖中文人名/地名，**实现时报告给作者**，不要自己加规则。

## 4. Generator 接口（与 Module 3 锁定）

### 4.1 RegressionPatternGenerator

文件：`core/anomaly/RegressionPatternGenerator.kt`

```kotlin
interface RegressionPatternGenerator {
    /**
     * 分析一条 USER_REPORT 类型的 Event，识别（或更新）一个回归值模式。
     *
     * @return AttentionAnomaly.id；如果未能识别出有意义的 pattern，返回 null。
     */
    suspend fun analyzeUserReport(eventId: String): Long?
}
```

**实现类**：`RegressionPatternGeneratorImpl`

**逻辑**：

```
1. 从 EventDao 取出该 Event（确认 type == USER_REPORT，否则抛 IllegalArgumentException）
2. 取最近 7 天的同 type AttentionAnomaly（用于 LLM 上下文 + 命中检查）
3. redactor.redact(event.text)
4. 调 LLM with BEHAVIOR_LINE_SYSTEM_PROMPT + 输入 schema (见 §5.1)
5. 解析输出 JSON：
   - patternType
   - patternTitle
   - patternDescription
   - severity (1-5)
   - confidence (0.0-1.0)
6. 计算 anomalyKey:
   - hash(patternType + 关键词摘要)；让"会议后回宿舍刷手机"和"开完组会回宿舍刷手机"命中同 key
   - 简化版：anomalyKey = "${patternType}:${normalize(patternTitle)}"
   - normalize: lowercase, 去掉标点和数字, 截断 50 字符
7. 命中检查：
   - SELECT * FROM attention_anomaly WHERE anomalyKey = ? LIMIT 1
   - 命中 → UPDATE: lastSeenAt=now, frequency=frequency+1
   - 未命中 → INSERT: status='ACTIVE', firstSeenAt=now, lastSeenAt=now, frequency=1
8. 返回该 anomaly 的 id；解析失败返回 null
```

**LLM 输出 JSON schema**（输出端）：

```json
{
  "patternType": "USER_REPORTED_DEFAULT_PATH",
  "patternTitle": "会议后低阻力恢复路径",
  "patternDescription": "用户开完会后疲劳，默认回宿舍刷手机。",
  "severity": 3,
  "confidence": 0.85
}
```

**校验**：
- `patternType` 必须在已知枚举中（见 `00_总览.md` §3.2 列表），否则用 `USER_REPORTED_DEFAULT_PATH`
- `severity` clamp 到 [1, 5]
- `confidence` clamp 到 [0.0, 1.0]
- 解析失败 → 返回 null（不落库）

### 4.2 AnomalySuggestionGenerator

文件：`core/anomaly/AnomalySuggestionGenerator.kt`

```kotlin
interface AnomalySuggestionGenerator {
    /**
     * 为指定 pattern 生成异常值建议，落库。
     *
     * @param patternId AttentionAnomaly.id
     * @param regenerationCount 第 N 次重试，0 = 首次
     * @param parentSuggestionId 如果是 POSTPONE 触发的重新生成，传入上一条 id；首次生成传 null
     * @return 新生成的 AnomalySuggestion.id；冷却中或失败返回 null
     */
    suspend fun generateForPattern(
        patternId: Long,
        regenerationCount: Int = 0,
        parentSuggestionId: Long? = null,
    ): Long?
}
```

**实现类**：`AnomalySuggestionGeneratorImpl`

**逻辑**：

```
1. 取 pattern：anomalyDao.getById(patternId)
2. 24h 冷却检查（仅对 regenerationCount == 0 生效；POSTPONE 重生成跳过）：
   - val recent = suggestionDao.findByPatternKeySince(pattern.anomalyKey, now - 24h)
   - 如果 recent.any { it.status in ["PENDING", "ACCEPTED", "REJECTED"] } → 返回 null
3. 取 7 天反馈历史：
   - val feedbacks = userFeedbackDao.recentByTargetType("SUGGESTION", now - 7days, limit=20)
   - 取每条对应的 suggestion，作为 prompt 上下文
4. 如果 parentSuggestionId != null：取 parent 的 actionText/whyText 作为"上一条"上下文
5. 调 LLM with BEHAVIOR_LINE_SYSTEM_PROMPT + 输入 schema (见 §5.2)
6. 解析输出 JSON（见 §5.3）
7. 三件套校验：
   - costLevel == "LOW"  → 不通过则重试 1 次
   - expectedUpside.isNotBlank() && length > 5
   - actionText.length <= 200, whyText.length <= 200
8. 校验失败 → fallback 静态文案
9. 落库 AnomalySuggestion(
       title = patternTitle (取自 pattern),
       suggestionDomain = "BEHAVIOR",
       actionText, whyText, costLevel="LOW", expectedUpside,
       status = "PENDING",
       patternId = patternId,
       patternKey = pattern.anomalyKey,
       regenerationCount,
       parentSuggestionId,
       modelVersion = "khup-behavior-v1",
       dayStartMs = todayStart,
       createdAt = now, updatedAt = now,
   )
10. 返回新 id
```

## 5. Prompt Schema

### 5.1 Pattern Generator 输入

User-role message JSON：

```json
{
  "user_report": "我刚开完会，平时会直接回宿舍刷手机",
  "current_time_local": "2026-05-09 21:14",
  "weekday": "Friday",
  "recent_patterns_7d": [
    {
      "type": "USER_REPORTED_DEFAULT_PATH",
      "title": "会议后低阻力恢复路径",
      "frequency": 3,
      "lastSeenDaysAgo": 1
    }
  ]
}
```

**Prompt 包装**：

```
SYSTEM: BEHAVIOR_LINE_SYSTEM_PROMPT
USER: 下面是用户提交的处境。识别这是一条什么类型的回归值模式（regression pattern）。
{
  "task": "identify_regression_pattern",
  "input": <上面的 JSON>
}

输出严格的 JSON：
{
  "patternType": "USER_REPORTED_DEFAULT_PATH",
  "patternTitle": "...",
  "patternDescription": "...",
  "severity": 1-5,
  "confidence": 0.0-1.0
}
```

### 5.2 Suggestion Generator 输入

```json
{
  "task": "generate_anomaly_suggestion",
  "pattern": {
    "type": "USER_REPORTED_DEFAULT_PATH",
    "title": "会议后低阻力恢复路径",
    "description": "用户开完会后疲劳，默认回宿舍刷手机。",
    "frequency": 3
  },
  "recent_feedbacks_7d": [
    {
      "suggestion_action": "去操场走 12 分钟",
      "feedback": "REJECTED",
      "reason": "下雨"
    },
    {
      "suggestion_action": "写下今天最强烈的念头",
      "feedback": "ACCEPTED"
    }
  ],
  "regeneration": {
    "is_regeneration": false,
    "previous_action": null,
    "regeneration_count": 0
  },
  "current_time_local": "2026-05-09 21:14",
  "available_minutes_estimate": 30
}
```

**当 `is_regeneration: true`**，prompt 多加一句：

> 用户对前一条建议点了"换一个"。前一条是："{previous_action}"。请给出**完全不同角度**的低成本异常值，避免重复。

### 5.3 Suggestion Generator 输出

```json
{
  "actionText": "不要立刻回宿舍。先在楼下走 10 分钟，期间不刷手机，只观察一个最想逃避的念头。",
  "whyText": "你现在不是单纯想休息，可能在寻找低摩擦情绪缓冲。这个小偏离能让你重新获得选择权。",
  "costLevel": "LOW",
  "expectedUpside": "打断会议后自动刷手机的惯性，恢复对疲劳状态的主动处理。",
  "tone": "sharp_but_not_shaming"
}
```

`tone` 字段当前不入库，留 future use。

## 6. JSON 解析的鲁棒性

LLM 输出经常带 markdown 代码块包裹。在 `core/ai/` 下加（如已有跳过）一个工具：

```kotlin
internal object LlmJsonExtractor {
    /**
     * 从 LLM 输出中提取首个 JSON 对象。
     * 容错：去掉 ```json ... ``` 包裹；找到第一个 { 到匹配的 }。
     * 失败返回 null。
     */
    fun extract(raw: String): String?
}
```

实现要点：
- `raw.replace("```json", "").replace("```", "")`
- 找到第一个 `{`，扫描配对的 `}`（忽略字符串内的括号）
- 用 `kotlinx.serialization` 或 `Gson` 反序列化

## 9. 模块验收

- [ ] `KhupPromptPolicy.BEHAVIOR_LINE_SYSTEM_PROMPT` 常量存在且文案符合三件套
- [ ] LLM logcat 不出现完整输出（grep `"LLM response"` 应只看到 redact 摘要）
- [ ] `RegressionPatternGenerator.analyzeUserReport` 单元测试：
  - mock LLM 返回有效 JSON → 返回 anomaly id
  - mock LLM 返回无效 JSON → 返回 null
  - 同 anomalyKey 重复调用 → frequency+1（不重复 INSERT）
- [ ] `AnomalySuggestionGenerator.generateForPattern` 单元测试：
  - 24h 内已有 PENDING → 返回 null
  - 24h 内已有 REJECTED → 返回 null
  - regenerationCount > 0 时跳过冷却
  - LLM 返回 costLevel != "LOW" → fallback 文案
  - LLM 抛异常 → fallback 文案
- [ ] 解析器能处理 markdown 代码块包裹的 JSON
- [ ] 文件长度：每个 Generator impl ≤ 300 行；超出请拆 helper

## 10. 边界提醒（实现 AI 不要做这些）

- ❌ 不要训练任何模型
- ❌ 不要把反馈聚合成"权重"或"评分"
- ❌ 不要把 LLM 错误暴露给用户（fallback）
- ❌ 不要添加"建议执行情况追踪"逻辑（违反不衡量执行原则）
- ❌ 不要在 prompt 里出现"完成率 / 打分 / 自律"等词
- ❌ 不要在本模块写 UI / Repository / DAO 增量；那些都在别的模块
